Areas de GIT

Working Directory

Es el area de trabajo, donde rara vez vamos a estar utilizando la linea de comandos, salvo por el inicio del proyecto o a lo sumo la eliminacion del mismo. Se refiere a la carpeta donde iniciamos el repositorio y basicamente donde vamos a crear, modificar y eliminar los archivos que formen parte de nuestro proyecto.

  • dir || ls muestra el contenido de la carpeta actual
  • mkdir <name> crea una carpeta con el nombre indicado
  • cd <path> abre el directorio especificado
  • git init inicializa un repositorio de git
  • rm -rf .git elimina el repositorio de manera forzada

Staging Area (INDEX)

El area de control de Cambios, corresponde a un archivo que se encuentra en la carpeta .git, y se utiliza para realizar las capturas de pantalla del codigo (snapshot) de los diferentes archivos del proyecto. Aqui se realiza el seguimiento de los mismos, para realizar la confirmacion y poder acceder a los diferentes cambios dentro de la historia de nuestro proyecto. Es importante tener en cuenta que todo archivo que no sea agregado al INDEX, no se podra recuperar en caso de ser eliminado.

  • git status muestra el estado de los archivos con respecto a la ultima confirmacion, en caso que exista.
    • untracked: al archivo no se le esta realizando seguimiento.
    • staged: los cambios fueron agregados, pero aun no estan confirmados
    • modified: el archivo difiere con respecto a lo almacenado en la ultima confirmacion
    • commited los cambios fueron almacenados en una confirmacion
  • git add <file> agrega al INDEX el/los archivo/s seleccionados
  • git rm --cached <file> quita del INDEX el archivo seleccionado.
  • git restore --staged <file> remueve los cambios de los archivos agregados al INDEX

Repository

Area donde se almacenan los cambios, mediante confirmaciones, en formato BLOB(Binary Large Object File). Solamente se confirman las modificaciones agregadas al INDEX; y mediante apuntadores podemos navegar dentro del historial.

  • git commit abre el editor para realizar la confirmacion
  • git blame <file> muestra linea por linea los autores de las confirmaciones, en caso que existan
  • git log muestra el historial de confirmaciones en orden descendente
  • git remote add <alias> <url> agrega una direccion remota, asignandole un alias
  • git fetch <remote> obtiene el historial de cambios de un repositorio remoto
  • git push <remote> <url> envia a un repositorio remoto los cambios

Apuntadores GIT

HEAD

Apuntador movil que permite al usuario desplazarse dentro del historial de cambios, ya que siempre indica la posicion actual del mismo. Tambien sirve como referente a la hora de indicar a que confirmacion estamos haciendo referencia al utilizar los comandos de git, como "HEAD^" para indicar que es un commit ancestro o "HEAD~2" para indicar que nos referimos al commit que se encuentra 2 confirmaciones atras.

BRANCH

Apuntador dinamico que siempre se corresponde con el ultimo commit de una historia, nos permiten volver cuando navegamos en el historial de cambios hacia un punto anterior y generalmente comparten un ancestro en comun. Se utilizan para implementar caracteristicas o corregir errores sin afectar la linea principal.

  • git branch <name>crea una rama en la posicion actual del historial
  • git branch -d <branch>elimina de forma segura una rama, siempre que los cambios hayan sido integrados
  • git branch -D <branch> elimina de manera forzada la rama, aunque los cambios no hayan sido integrados
  • git push -u <remote> <branch>agregamos una rama local a nuestro repositorio remoto
  • git push -d <remote> <branch>elimina del repositorio remoto la rama seleccionada
  • git merge <branch> integra en la rama actual los cambios de la rama seleccionada.

TAG

Apuntador estatico que siempre se corresponde con un commit determinado, se utiliza principalmente para el versionado del proyecto y nos permite desplazarnos mucho mas facilmente en el historial de cambios. Normalmente se utiliza una estructura similar a v.n.n.n donde se especifica el numero de version principal, compilacion o subversion y correcciones o revisiones.

  • git tag <name> crea una etiqueta sobre el commit actual
  • git tag -l lista las etiquetas asignadas localmente
  • git tag -d <tag> elimina la etiqueta seleccionada
  • git tag <old> <new> agrega una etiqueta en la misma posicion que la seleccionada
  • git push <remote> --tags envia las etiquetas al repositorio remoto

STASH

Apuntador que hace referencia a un commit y su INDEX en un punto determinado dentro del historial. Se conoce tambien, como la zona o pila temporal de cambios. Se utiliza para desplazarnos dentro del historial de cambios, cuando tenemos cambios pendientes en el INDEX y estos no ameritan un commit. Al reaplicar los cambios se realiza una operacion merge a la inversa.

  • git stash agrega los cambios del INDEX a la zona temporal.
  • git stash apply @stash{n} aplica los cambios de la zona temporal
  • git stash drop @stash{n} elimina las referencias de la zona temporal
  • git stash pop @stash{n} aplica los cambios y los elimina de la zona temporal
  • git stash list lista las referencias de la zona temporal

Control de Cambios

RESET

Comando que permite la eliminacion de commits del historial, reestablecer el proyecto a un punto anterior o simplemente deshacer los cambios del INDEX. Este comando elimina desde el ultimo commit al seleccionado. Es bastante practico cuando necesitamos deshacer cambios, siempre y cuando estos no hayan sido subidos al respositorio remoto.

  • git reset --soft <ref> Valor predeterminado, que solamente elimina los commits sin reestablecer el proyecto.
  • git reset --mixed <ref> Elimina los commits y reestablece los cambios del INDEX a dicho punto de control.
  • git reset --hard <ref> Reestablece el proyecto completo hasta el commit seleccionado.
  • git reset <ref> -- <file> Reestablece un archivo a como estaba en el commit referenciado.
